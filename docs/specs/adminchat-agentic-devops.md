# Спецификация: агентный девопс-исполнитель adminchat

Статус: проект. Решение ADR-0041. Переопределяет узкий каталог действий ADR-0037/0038,
сохраняя гарды против зацикливания, аудит и жизненный цикл инцидентов.

## 1. Обзор

Панель adminchat получает настоящего ИИ-агента, который использует инструменты циклом
«наблюдение, рассуждение, объяснение, действие, проверка». Модель (Gemma через vLLM,
LLM_SERVICE_URL) формулирует произвольные команды под ситуацию и объясняет действия по-русски.
Команды исполняются на кластере (kubectl через API) и на узлах (через node-agent). По умолчанию
автономно; подтверждение только для классов finance и destructive; переключаемый ручной режим.

## 2. node-agent (привилегированный DaemonSet)

Отдельный сервис services/node-agent (маленький HTTP-сервер на Python stdlib, без внешних
зависимостей). Манифест deploy/k3s/99-node-agent.yaml: DaemonSet, привилегированный
(`securityContext.privileged: true`), `hostPID: true`, `hostNetwork: true`, монтирует хост в
режим исполнения в пространстве хоста через `nsenter -t 1 -m -u -i -n -- <argv>`. Слушает порт
9110 только внутрикластерно (без Ingress, без Service типа LoadBalancer; обычный ClusterIP или
обращение по IP пода/узла). Toleration на все taint-ы (в том числе GPU), чтобы под был на каждом
узле.

API:
  `GET  /health` -> `{"status":"ok","node":"<имя>"}`.
  `POST /run` (заголовок `X-NodeAgent-Token: <NODEAGENT_TOKEN>`), тело
    `{"argv":["df","-h","/"],"timeout":30}`; исполняет argv в пространстве хоста; возвращает
    `{"exit_code":int,"stdout":str,"stderr":str,"duration_ms":int,"node":str}`. Ввод строго argv
    (список), НЕ строка оболочки: инъекция оболочки исключена, `nsenter` вызывается без `sh -c`.
    Ограничение вывода (например 256 КБ), обрезка с пометкой. Таймаут обязателен.
  Fail-closed без токена (401). Токен из секрета `nodeagent-secrets`, общий с adminchat.

Панель обращается к node-agent конкретного узла. Discovery: `GET /nodes` у самой панели строит
карту узел -> адрес пода node-agent (через Kubernetes API, endpoints DaemonSet), либо панель
шлёт на node-agent любого узла запрос с полем `node`, а тот при несовпадении имени возвращает
ошибку (проще: панель адресует под нужного узла по его podIP из Kubernetes API).

## 3. Инструменты агента (в adminchat)

Новый модуль services/adminchat/agent_exec.py: цикл использования инструментов. Модель получает
системную инструкцию и набор инструментов, на каждом шаге возвращает СТРОГО JSON одного вызова
(схема проверяется кодом, как в solve.py), панель исполняет, возвращает результат модели, и так
до финального ответа или лимита шагов (AGENT_MAX_STEPS, по умолчанию 12).

Инструменты:
  `observe`  {argv, target}  где target = "control"|"gpu"|"cluster". Read-only команда (kubectl
     get/describe/logs/top, df, du, docker ps, crictl images, ps, free, uptime, cat /proc/...).
     Всегда разрешена, в бюджет не считается. Классификатор обязан подтвердить, что команда
     read-only (список безопасных бинарей и подкоманд); иначе она идёт как `act`.
  `act`      {argv, target, why}  Мутирующая команда (kubectl delete/patch/rollout/scale, docker
     system prune, crictl rmi, rm, kill, systemctl restart, освобождение диска). Проходит через
     политику раздела 4.
  `node_cmd` {node, argv, why}  Команда на конкретном узле через node-agent. Тоже через политику.
  `explain`  {text}  Промежуточное или финальное объяснение оператору на русском.
  `done`     {summary}  Завершить задачу с итогом.

Каждый шаг агента (мысль, вызов, результат) стримится в чат панели как системные строки, чтобы
оператор видел ход рассуждения и команды в реальном времени.

## 4. Классификация и политика (детерминированная, вне модели)

Модуль services/adminchat/policy.py, чистые функции плюс тесты. Функция `classify(argv)` относит
команду к одному из классов:
  `read`         только чтение (см. список безопасных). Автономно всегда.
  `safe_write`   ремонт без разрушения данных: rollout restart, delete pod, requeue, docker/crictl
     prune, rm кешей/временных, kill процесса, systemctl restart, освобождение места, изменение
     флагов рантайма. Автономно в auto-режиме; предложение в manual.
  `finance`      тарифы, баланс, платежи, деньги тенанта (пути /api/admin/tariff, grant-minutes,
     reapply, renew-plan, extend-storage, любые изменения minute_packs, subscriptions, payments).
     ВСЕГДА подтверждение, в любом режиме.
  `destructive`  необратимое: rm -rf на данных, drop/truncate таблиц, удаление PVC и томов, wipe
     файловой системы, удаление тенанта или пользователя, kubectl delete namespace/pvc/pv.
     ВСЕГДА подтверждение. Неоднозначное относится сюда (fail-safe).

Классификатор строится на списках паттернов (бинарь плюс аргументы, целевые ресурсы Kubernetes,
опасные флаги). Список destructive и finance это критичный код с тестами, любое дополнение через
тест. По умолчанию неизвестная мутирующая команда трактуется как destructive.

Политика исполнения: в auto-режиме read и safe_write исполняются сразу; finance и destructive
идут в карточку подтверждения (существующий механизм PENDING/execute_confirmed, усиленное
подтверждение для destructive вводом контрольного значения). В manual-режиме любая мутирующая
команда (safe_write, finance, destructive) идёт оператору на выбор или ввод своей команды.

## 5. Режимы

Флаг режима в runtime_flags (или локально у панели), команда `/mode auto|manual` и в /agent.
  auto (по умолчанию): агент исполняет сам по разделу 4.
  manual: агент на каждом шаге, где требуется мутация, вместо исполнения возвращает
     пронумерованные варианты с риском и рекомендацией; оператор отвечает номером, либо вводит
     свою команду/инструкцию на естественном языке (агент её интерпретирует и исполняет через ту
     же политику). Наблюдение (observe) агент делает сам и в manual, чтобы собрать данные.

Свободная команда оператора: в чате оператор может в любой момент дать инструкцию («почисти диск
на gpu-узле», «покажи per-core загрузку cpu», «убей зависший процесс X», «перезапусти asr») или
прямую команду; агент планирует и исполняет по политике.

## 6. Гарды и аудит (сохраняются из ADR-0038)

Все мутации (act, node_cmd, finance, destructive) проходят через guards.py: бюджет действий в
час, кулдаун на повторное одинаковое действие, предохранитель после серии неудач, детектор
осцилляции. Каждое исполнение пишется в audit.py и в платформенный audit_log (где применимо) с
actor=agent или actor=<оператор>, полным argv, узлом, кодом возврата и обрезанным выводом.
Read-only в бюджет и кулдауны не входят.

## 7. Тише лента инцидентов

Порог регистрации инцидентов поднимается: поллер заносит инцидент только при band=high ИЛИ при
подтверждённом детекторе застревания/недоступности, но НЕ при band=uncertain на общем анализе
логов. Нормальные состояния («застрявших нет, очередь движется») инцидентами не становятся.
Одинаковые по отпечатку продолжают группироваться со счётчиком (уже есть). Добавить команду
`/incidents purge-noise`, помечающую разрешёнными шумовые группы по эвристике (нулевой band,
нет затронутых тенантов, детекторы только уровня note).

## 8. Эндпоинты панели

  `POST /agent/run` {message}  запустить агентный цикл над инструкцией оператора; стримит шаги.
  `POST /agent/confirm` {token}  подтвердить отложенную finance/destructive команду.
  `POST /agent/mode` {mode}  переключить auto|manual.
  `GET  /agent/state`  режим, бюджет, кулдауны, последние действия, предохранитель.
Существующие /chat, /incidents* сохраняются; /chat в свободном диалоге теперь может передавать
управление агентному циклу, если оператор просит действие.

## 9. Порядок внедрения

Этап 1: node-agent (DaemonSet, API, секрет) и его выкатка на оба узла; проверка `df`, `docker
system df`, `nsenter` работают на хосте.
Этап 2: policy.py (классификация read/safe_write/finance/destructive) с тестами; agent_exec.py
(цикл инструментов) с фолбэком без модели; эндпоинты /agent/*; режимы.
Этап 3: интеграция в панель (стрим шагов, карточки подтверждения finance/destructive, /mode,
свободная команда оператора).
Этап 4: подъём порога инцидентов и purge-noise.
Этап 5: боевая выкатка и проверка на реальных задачах (диск gpu-узла, объяснение per-core CPU,
таймлайн инцидентов).
